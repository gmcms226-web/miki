### 1. 컴포넌트 기반 구조

디바운스를 적용한 검색 로직
function useDebouncedSearch(callback, delay = 300) {
  const timer = useRef(null);

  return (value) => {
    clearTimeout(timer.current);
    timer.current = setTimeout(() => callback(value), delay);
  };
}

const [page, setPage] = useState('home')

## 11. Trouble Shooting

| 문제 | 원인 | 해결 |
|---|---|---|
| 이미지가 보이지 않음 | 파일명 대소문자 또는 경로 불일치 | 실제 파일명 확인 후 경로 수정 |
| 레이아웃 깨짐 | 고정 너비와 반응형 조건 충돌 | 미디어쿼리 및 grid/flex 조정 |
| 새로고침 시 데이터 초기화 | 상태값만 사용 | 필요한 데이터는 localStorage 저장 |
| 배포 환경 오류 | 환경 변수 누락 | Vercel 환경 변수 등록 |
| 버튼 클릭 미동작 | 실제 섹션 id와 링크 불일치 | 앵커 및 이벤트 핸들러 수정 |

## 18. 프로젝트 회고


이번 프로젝트를 React로 클론코딩하면서 컴포넌트 단위로 화면을 쪼개고, 상태를 관리하며 실제 서비스의 흐름을 재현하는 경험을 할 수 있었습니다. 단순히 정적인 화면을 옮기는 것이 아니라 로그인/회원가입, 장바구니 담기, 관심 상품 등록, 결제 같은 실제 사용자 플로우를 하나씩 구현하면서 React의 컴포넌트 구조와 상태 흐름을 체감할 수 있었습니다.

- **로그인 / 회원가입**: Firebase Authentication을 연동해 실제 인증 흐름을 구현하며, 폼 유효성 검증과 에러 처리, 로그인 상태에 따른 라우팅 분기를 직접 다뤄봤습니다.
- **장바구니 담기**: 상품 담기/수량 변경/삭제 로직을 상태로 관리하면서, 여러 컴포넌트에서 동일한 장바구니 상태를 공유하기 위한 전역 상태 설계의 필요성을 느꼈습니다.
- **관심 상품 등록**: 찜하기 토글 UI와 사용자별 관심 목록 저장을 구현하며, Firebase의 데이터 구조를 사용자 단위로 설계하는 경험을 했습니다.
- **결제**: 장바구니 데이터를 기반으로 주문 정보를 구성하고 결제 프로세스를 연결하면서, 여러 단계(장바구니 → 주문 확인 → 결제)를 넘나드는 상태 유지의 어려움을 직접 마주했습니다.

작업 과정에서 상태가 여러 컴포넌트에 걸쳐 꼬이거나, 새로고침 시 로그인/장바구니 상태가 초기화되는 문제 등을 겪으며 상태관리와 데이터 영속성에 대한 고민을 많이 했습니다. 이 과정에서 React 개발자 도구와 브라우저 콘솔을 적극 활용해 상태 변화를 추적하는 습관을 기를 수 있었습니다.

앞으로는 전역 상태관리 라이브러리 도입, 결제 연동의 예외 처리 강화, 테스트 코드 작성 등을 통해 더 견고한 서비스로 확장해보고 싶습니다.

- **잘한 점**: 로그인부터 결제까지 이어지는 실제 사용자 플로우를 컴포넌트 단위로 나누어 구현
- **아쉬운 점**: 전역 상태관리 구조를 초기에 설계하지 못해 장바구니/로그인 상태 관리가 복잡해진 점
- **다음 프로젝트에 적용할 점**: 상태관리 라이브러리 도입 및 결제/인증 관련 예외 처리 강화
